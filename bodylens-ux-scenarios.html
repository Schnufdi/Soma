Session management — 4 scenarios
01
Missed yesterday's session — want to do it today
The most common use case for any training app
Full pass
User intent
Missed Monday's push session. It's Tuesday. I want to do it today instead of my planned pull.
What they do
Opens daily plan. Sees Monday's chip highlighted red (missed). Clicks it. Recovery modal appears — shows the missed session type, offers available slots, defaults to today.
System response
Works fully. openRecoveryModal() reschedules the session by writing a week override to localStorage. The missed day is marked skipped. The page reloads showing today as a training day with the recovered session. Cache is busted automatically.
Friction
The missed-day chip is 8px, hard to tap on mobile. The entry point is not labelled. A new user wouldn't know to click the red dot.
What's good
The underlying recovery logic is excellent — it correctly detects missed sessions, offers valid slots only, marks the original day, and reloads with the new plan. This is genuinely rare functionality in health apps.
What to fix
Make the missed-day chip larger and add a prompt: "You missed Monday — tap to reschedule." Surface it proactively in the daily plan header, not buried in a tiny dot.
02
Shoulder's sore — want to swap bench for dumbbell press
Mid-session exercise substitution
Full pass
User intent
About to start push day. Left shoulder is flaring. Want to replace barbell bench with dumbbell press and reduce load.
What they do
Opens the training block. Taps the Session Editor. Sees intensity controls (beginner/intermediate/advanced) and volume selector (3–6 sets). Changes intensity to lighter scheme. The exercise list updates accordingly.
System response
Works for volume and intensity. lockSessionIn() mutates the exercise list and writes to bl_daylog_[DATE].sessionEditor. Changes persist and affect the day score.
Friction
The session editor doesn't let you pick a specific replacement exercise by name — you change intensity scheme, not individual lifts. If you specifically want "dumbbell press instead of bench", you can't select that. You have to log the swap as a free-text note with the ~ button.
Gap
"Swap this specific exercise" is not possible. The editor changes the whole session scheme. A true exercise-level swap (bench → dumbbell press) requires the user to manually mark the bench as modified and type what they did instead. The coach won't see this as an explicit swap — it reads as a note.
03
Travelling — no gym, need a bodyweight session
Plan needs to adapt to zero equipment
Full pass
User intent
I'm in a hotel in Amsterdam. No gym. I still want to train. Give me a bodyweight session that maps to my programme.
What they do
Opens the daily plan on a training day. Sees the ✈ Travelling? Swap to bodyweight → chip in the mission header. Taps it. A modal slides up showing the current session type and the bodyweight alternative. Taps "Confirm travel mode."
System response
Works fully. openTravelModal() detects the session type (push/pull/posterior chain/upper/full body) and maps it to a curated bodyweight exercise set via _travelBWWorkout(). The training block is mutated in place — exercises, subtitle, coach note all replaced. The swap is persisted to bl_daylog.travelMode and to the plan cache. Original session is stored for restoration. Cancel mode fully reverts.
Friction
Travel mode is session-scoped: it modifies today's block but doesn't write a multi-day override (so tomorrow still shows the gym session). User must re-activate each day. Acceptable for a first pass.
What's good
The chip is visible directly in the mission header on every training day — no hunt through settings. Cancel mode correctly restores the original exercises from the saved log data. The bodyweight alternatives are programme-type-aware (push day → press/dip/pike push-up, posterior chain → glute bridge/Nordic curl/back extension).
What to improve next
Multi-day travel mode: a toggle that persists across days without needing to re-activate. And post-session: "Rate this bodyweight session" to build data on bodyweight tolerance for the reconciliation pass.
04
Only have 30 minutes — session is planned for 60
Time-constrained day, need a condensed session
Full pass
User intent
Unexpected meeting. I can only do 30 minutes. What's the highest-value 30-minute version of today's session?
What they do
Two paths: (a) Tap the ⏱ Short on time? chip in the mission header — opens the session editor with the time selector scrolled to the top. (b) Open the session editor directly. At the top: time selector chips 30 / 45 / 60 / 90 min.
System response
Works fully. Selecting 30 min auto-sets volume to 3, strips cardio, clears cardioDuration/kcal. 45 min sets volume 4 + no cardio. 60 min sets volume 5. 90 min sets volume 6. The contextual entry point (openShortTimeFlow()) is surfaced in the mission header only when the session isn't done — shows at the right moment, disappears once it's irrelevant.
Friction
Manual volume override still clears timeBudget — so if you pick 30 min then manually bump volume, the time-budget chip deselects (expected behaviour but slightly surprising). Minor edge case.
What's good
The "Short on time?" chip is contextual — only appears when the session isn't yet done, so it's visible exactly when needed and gone when not. The time selector is the first element in the session editor body, so the user lands on it immediately via openShortTimeFlow(). No hunting.
Nutrition scenarios — 4 scenarios
05
Ate something completely different for lunch
Log actual food vs planned meal
Partial
User intent
Plan said chicken and rice. I had a burrito at a work lunch. I want to log what I actually ate so macros stay accurate.
What they do
Tap the ~ (swap) button on the lunch block. A text field appears. They type "chicken burrito — approx 650 kcal, 35g protein." This is saved to the day log. Alternatively, open the voice guided log and describe the meal by voice.
System response
Logs the swap text, doesn't parse macros. dpSaveSwap() stores the free-text note. The macro tracker doesn't update — the protein bar still shows the planned value. The coach review at end of day will read the swap note, but the live macro display remains wrong.
User experience
They logged something but the numbers don't change. Feels incomplete. The macro bar's inaccuracy makes the whole tracking loop less trustworthy.
Critical gap
The swap logs text but doesn't update macros. To update macros, the user must go to the manual macro entry (if it exists) separately. These two flows are disconnected. The swap note should trigger a macro re-estimate — even a rough one from Claude ("estimated 35g protein from your note").
06
6pm — 80g protein short. What do I eat tonight?
Real-time adaptive nutrition guidance
Partial
User intent
It's 6pm. I check the macro bar. I'm 80g short on protein. I need Soma to tell me what to eat for the rest of the day to close the gap.
What they do
SOS: the macro shortfall is automatically visible to the SOS system — it reads remaining macros before building the prompt. Opens SOS or asks the coach "I'm 80g short on protein, what should I eat tonight?"
System response
SOS is aware of remaining macros. The prompt includes remaining.prot — Claude knows the shortfall and gives specific meal suggestions. Coach chat also has access. The response is good quality but advisory — the plan doesn't update.
Friction
User has to actively ask SOS or the coach. There's no proactive alert: "You're 80g short on protein at 6pm — here's what to eat." The data exists, the trigger doesn't.
What to add
An evening nutrition nudge: if it's after 5pm and the user is >40g short on protein, show a contextual banner: "You're [X]g short on protein — tap for ideas." One check on page load, one banner, one tap to SOS. Uses data already in MACROS global.
07
Ran out of creatine — want to pause that supplement
Stack management mid-programme
Not built
User intent
I've run out of creatine. I don't want to reorder for 2 weeks. I want to remove it from my daily stack checklist until it arrives.
What they do
There is no "pause supplement" control in the daily plan or supplement page. The creatine remains on the stack list. The user marks it as skipped every day. After 3 days this becomes friction and they stop logging the stack entirely.
System response
No mechanism. Supplements are set during onboarding and editable only via the supplement stack editor. There's no per-supplement pause or "out of stock" flag.
Downstream effect
Daily stack completion score drops. Debrief shows low supplement adherence. Coach may flag supplement compliance as an issue. All because of an empty tub.
What to build
A hold/pause toggle on each supplement in the stack view. "Pause for [1 week / 2 weeks / until I tap again]." During a pause, that supplement is hidden from daily checklist and excluded from adherence score. Two-tap fix, prevents a downstream data quality problem.
08
IBS flare-up today — does the meal plan adapt?
Health condition — acute episode, real-time plan change
Not built
User intent
I have IBS in my health profile. I'm having a bad gut day. I want today's meals to go low-FODMAP and training to be lower intensity.
What they do
There is no "bad gut day" flag. The user opens SOS — there's no IBS-specific scenario. They could type "IBS flare today" in free text. Claude gives advice. The meal plan doesn't change. Today's training plan doesn't change.
System response
IBS exists in the risk classifier for general coaching, but there's no acute episode mode. Health conditions in the profile inform the weekly meal plan and coach system prompt, but don't create day-level overrides.
Gap severity
High. Users with chronic conditions have bad days. If the app ignores those days, the plan becomes actively harmful (eating high-FODMAP on a flare day, doing HIIT when gut motility is elevated).
What to build
A "Today I'm…" status chip at the top of the daily plan: [Tired] [Sick] [Gut flare] [Sore] [Travelling] [Low energy]. Selecting one triggers a plan overlay that modifies today only — drops intensity, swaps high-FODMAP meals, adjusts timing. One tap, immediate adaptation. Session state, not profile state — resets tomorrow.
Daily flow scenarios — 5 scenarios
09
Terrible sleep last night — should today's training change?
Sleep-adaptive programming
Full pass
User intent
Slept 4 hours. Feel terrible. My programme says heavy squats today. Should I still do it or should Soma adapt?
What happens
On page load, checkSleepIntensityBanner() reads bl_daylog_[YESTERDAY].sleepActual. If it's below 6 hours and today is a training day: an amber banner injects above the plan — "You logged [X]h sleep last night. Consider reducing today's intensity." One tap: "Reduce to Foundation intensity →"
System response
Works fully. applySleepIntensityReduction() sets _seState.intensity = 'beginner' and opens the session editor pre-loaded with reduced intensity. Seen-key bl_sleep_banner_[DATE] ensures the banner only shows once per day. Skipped if intensity already reduced today. Sleep must be actively logged the previous day to trigger.
What's good
No other consumer health app does this — reads yesterday's logged sleep and surfaces an actionable offer the next morning. One tap delivers the adaptation. The banner is non-blocking: if the user ignores it, the rest of the plan loads normally.
What's good
The data loop is coherent: sleep is logged in EOD card → read next morning → adaptation offered. Three sleep logging paths (EOD card, 8pm evening bar, day-close card) increase the likelihood that sleep data is actually captured.
What to improve next
Trigger also on energy score ≤ 2 from the previous day's log, not just sleep hours. Sleep is the primary signal, but logged energy rounds it out for cases where sleep was 6h but quality was poor.
10
End of day — closing the loop properly
Day completion flow
Full pass
User intent
Done for the day. I trained, ate well, hit most of my macros. I want to log everything and get a coach summary of how I did.
What they do
Option A: Tap each item ✓ as they complete it through the day. At the end, tap "Save & close the day." Option B: Open the voice/guided debrief — AI asks questions about training, sleep, nutrition via conversation. Auto-extracts structured data. Option C: SOS or coach chat for the review.
System response
Works well. All three paths store data in bl_daylog_[DATE]. The voice debrief is genuinely impressive — Claude extracts exercise weights, protein estimate, energy level, sleep quality from natural conversation and commits structured JSON. Coach review generates a specific plan vs actual analysis.
Friction
The EOD card is buried at the bottom of a long page. Users who don't scroll to the end won't close the day. The debrief is powerful but not obviously discoverable — it requires tapping a non-labelled "Log day" button.
What's good
Three completly different logging paths (tap-by-tap, voice guided, text coach) all converging on the same structured data. This is thoughtful — different users will use different paths.
What to improve
Show a sticky "Close today →" bar once it's past 7pm and the user has logged at least one item. Brings the EOD action into view without requiring scroll.
11
Mid-session — exercise is causing sharp pain
In-session injury response
Partial
User intent
I'm in the gym. On set 2 of Romanian deadlifts, I feel a sharp pull in my left hamstring. I need to stop this exercise and know what to do.
What they do
Tap × (skip) on the exercise. The block is marked skipped. Then open the Gym Coach overlay, which is a live step-by-step session guide. Or open SOS and type "sharp pull in left hamstring during RDLs."
System response
SOS handles this well — gives immediate protocol advice. The Gym Coach is a live guided workout flow (full-screen immersive overlay) but doesn't have injury-response branching. There's no "flag this exercise as injured" flow that persists to the profile's injury list.
Critical gap
The injury is not written to the profile. Tomorrow's plan will include Romanian deadlifts again. Next week's plan will include them. The system has no memory of what happened today. The user must manually go to the profile editor and add a hamstring injury — which they won't do when they're in pain at the gym.
What to build
When a user skips a block with the × button and then opens SOS, offer: "Add this to your injury log?" One tap saves it to profile.injuries. Next day's plan builder sees it. This is the difference between a programme that adapts and one that just displays.
12
Forgot to log yesterday — can I go back?
Retroactive logging
Full pass
User intent
It's Tuesday morning. I forgot to log Monday's workout. Can I go back and log it so the weekly data is accurate?
What they do
Tap Monday's chip in the prog-banner. The plan shows Monday's session. The session editor, debrief, and logging buttons are all still functional on past days. They mark the session done and log their weights.
System response
Works. jumpDay(i) calls getDayCtx(p, i) → buildPlan(p, dayCtx) → renderPlan(plan, true). The browsed day's correct block structure renders (training vs rest, correct exercises). Status buttons write to bl_daylog_[MONDAY DATE]. The weekly data is updated retroactively.
Friction
The day chips are small — the main navigation discovery issue. But once tapped, browsing is reliable. A "← Back to today" button renders at the bottom of every browsed day for clear return navigation.
Fix applied
A silent regression had caused renderPlan to always use TODAY.isTraining instead of the browsed day's training flag — so browsing a training day from a rest day rendered the wrong act structure and the training blocks simply didn't appear. Fixed: isTraining is now derived from !!trainBlock (the actual plan content) when browsing. Log is also cleared to {} on browse so today's completion state doesn't bleed in.

Why am I doing Romanian deadlifts? Explain this to me.
In-context exercise rationale
Full pass
User intent
I see RDLs on my plan. I've never done them properly. I want to understand why I'm doing them, how to do them, and what they're working.
What they do
Tap to expand the training block. The expanded detail shows the session logic (why this session type today), exercise list with form cues in the detail text, and a "Review with coach →" button that opens the AI coach.
System response
Works well. Expanded block shows session reasoning and exercise context. Coach chat is available with full profile context including the current day's plan — responds specifically about today's exercises in the context of the user's goal, experience level, and injury history.
What's good
The page-level context injection in coach.js means the coach answers about RDLs in the context of this user's fat loss goal, intermediate experience, and existing knee injury — not generically.
Weekly scenarios — 4 scenarios
14
End of week — does Soma review and re-plan?
Weekly reconciliation and adaptation loop
Partial
User intent
It's Sunday. I want to know how my week went, what patterns the system noticed, and how next week should differ.
What happens
dp-reconcile.js runs checkAndRunReconciliation() once per week (keyed by ISO week number). If 3+ days of logs exist, it sends a 7-day summary to Claude. Claude returns JSON with weekRating, patterns[], adaptations[], nextWeekFocus. This is saved to bl_profile.weeklyAdaptation and displayed as a coaching card.
System response
Analysis works, adaptation doesn't auto-apply. The reconciliation generates genuinely useful insights — "You consistently undertrain on Thursdays", "Energy is low Tuesdays after Monday night alcohol." But the adaptations (e.g., move Thursday to Friday, reduce volume week 3) are advisory text only. The user must manually edit remaining days, which most won't.
Gap severity
This is the core intelligent feedback loop of the product. It exists as analysis but not as action. The difference between Soma being a sophisticated plan display and a genuinely adaptive system lives here.
What to build
"Apply these adaptations to next week?" — a single confirm button that takes the adaptations[] JSON from the reconciliation result and writes them as week overrides. This is the highest-leverage unbuilt feature in the entire product. The AI already produces the right output — it just isn't being applied.
15
3 missed sessions this week — does the system flag?
Compliance monitoring and intervention
Full pass
User intent
Tough week. Missed 3 of 4 training sessions. I want the app to notice, not judge, and help me figure out what to do.
What happens
The week banner flags "behind" status. Missed day chips turn red. Opening the recovery modal when ≥ 2 sessions are missed activates Salvage Week mode.
System response
Salvage week mode works. When missedSessions.length ≥ 2, the modal sorts sessions by recency, schedules only the 2 most recent (the highest-value remaining), and silently writes off the rest. A "Compensation hacks" section appears: no alcohol, extra protein, cold shower, walk, sleep, deload next session — simple recovery actions the user can commit to. Written-off sessions don't add to the reschedule burden.
What's good
The UX correctly distinguishes "1 missed session" (full recovery modal, pick any slot) from "2+ missed sessions" (salvage mode, don't overwhelm — pick the 2 best, let the rest go). This matches how a real coach would respond: not "reschedule everything" but "here's what's salvageable."
What's good
Compensation hacks are contextual and actionable — not generic wellness advice. They're framed as commitments the user can tap to select, and saved to daylog.salvageCommitments for reference in the weekly reconciliation.
What to improve next
Surface salvage commitments in the following day's plan (e.g. "You committed to extra protein today — your morning meal target is highlighted"). Closes the loop from commitment to action.
16
Completed a great week — does it build on it next week?
Progressive overload and programme evolution
Full pass
User intent
I logged every session, hit all my macros, felt great. I want next week to progress — more volume, more weight.
What happens
The 4-week programme generated at onboarding has a weeklyProgression object — each week has a defined phase (Foundation → Build → Peak → Deload) with volume and intensity notes. Week 2 loads week 2's progression automatically by calculating days since generatedAt. Exercise weights increase based on logged weights from bl_ex_weights.
System response
Works. Progression is built into the 4-week structure. Exercise loads adapt based on logged weights. The system tracks "did this user actually do the planned volume?" via weekly reconciliation and can recalibrate if consistently over/under-performing.
What's good
The bl_ex_weights key — which stores per-exercise logged loads — means that over time, the programme accumulates actual performance data. This is the data model for genuine progressive overload.
17
6 weeks in, I feel like I need a deload
User-triggered deload week
Not built
User intent
I've trained hard for 6 weeks. Everything aches. I want Soma to give me a deload week — reduced volume, same frequency, active recovery focus.
What happens
There's no deload trigger. Week 4 of the generated programme is a deload phase, but once the 4-week block ends there's no continuation logic. The system doesn't auto-detect overtraining from accumulated fatigue scores. The user opens SOS or asks the coach — gets advice — but can't click "give me a deload week" and have it happen.
System response
Not implemented. Deload is mentioned in reconciliation advice text, but there's no mechanism to apply it. The user is left reading advice and having to either manually log every session as "lighter" or restart their programme.
What to build
A "Request deload week" button in the week banner or session editor. When triggered: (1) fetches today's week plan, (2) generates a deload version via API (50–60% volume, same exercise pattern, RPE 6 max), (3) writes as week overrides for the remaining days. Resets automatically the following Monday. The reconciliation already suggests deload when warranted — the missing piece is the one-tap application.
Programme lifecycle scenarios — 4 scenarios
18
End of 4-week block — what happens on Day 29?
Programme lifecycle and reassessment
Not built
User intent
I've completed my 4-week programme. I expect Soma to assess my progress, update my numbers, and generate a new 4-week block that's more advanced.
What happens
Day 29 loads. getProgCtx() still returns weekNum: 4, phase: 'Deload' (capped at 4). The prog-banner shows "WK4 · DELOA." The plan generates identically to Day 28. No reassessment prompt. No new block. The user's programme is stale.
System response
Silent failure. The 4-week tracking exists in the code (days 1–28), but there's no boundary event at Day 29. The programme never prompts reassessment, never generates a new block, and the progress tracker silently loops at Week 4.
User experience
User opens Soma on Day 29 and sees exactly the same plan as yesterday. Nothing acknowledges their 4-week achievement. The app gives no sense that they've completed a block. For a subscription product, this is a churn moment — the user feels forgotten.
What to build — highest priority
A Day 28 completion screen: acknowledge the block, summarise performance from 28 days of logs, prompt check-in (weight, how you feel, goal update), and generate a new 4-week block using updated data. This is the "graduation moment" that creates long-term subscriber retention. Without it, the programme becomes stale after one block — which for most users is weeks 5–8, exactly the high-risk churn window.
19
New injury mid-programme — knee starts hurting
Profile update → programme adaptation
Partial
User intent
Three weeks into my programme, I develop a knee problem. I need to update my injury status and have the remaining programme adapt — no squats, no lunges, safe alternatives.
What they do
Go to Profile → Health & conditions → Edit → add Knee to injuries. This now works thanks to the new profile editor. Saves to bl_profile.injuries. Return to the daily plan.
System response
Profile update works (new feature). Plan adaptation is partial. The coach system prompt now includes the knee injury. The risk classifier will flag knee-risk exercises. But today's cached plan still contains squats — the plan cache isn't busted when the profile is updated.
The gap
Profile update ≠ plan update. The daily plan reads from its cache, not live from the profile on every render. A profile change won't reflect in today's plan unless the user manually rebuilds ("↺ Rebuild plan").
What to fix
When the profile's injury or healthConditions fields change, bust the day plan cache. One line in the profile save function: localStorage.removeItem('dayplan_v6r3_' + TODAY). The next load auto-regenerates with the updated injury context. This closes the feedback loop between profile edit and plan content.
20
Goal change — shifting from fat loss to muscle building
Mid-programme goal pivot
Partial
User intent
I've lost the weight. Now I want to build muscle. I want to tell Soma my goal has changed and have the calorie target, protein target, programme split, and coach framing all update.
What they do
Profile editor → Goals → change to "Build muscle." This now saves to bl_profile.goal. The coach immediately knows the new goal (reads from profile). The science pages update their callouts. Meals page generates muscle-building meal plans from next generation.
System response
Coach and content update immediately. Macros and programme do not. The calorie target, protein target, and weekly training split were generated at onboarding for fat loss. Changing the goal field doesn't recalculate macros or regenerate the programme structure.
User experience
The coach is now telling them to build muscle while the daily plan has them in a caloric deficit doing fat-loss volume. The advice and the plan are contradicting each other.
What to build
When goal changes in the profile editor, trigger a lightweight macro recalculation (calories, protein, carbs, fat) based on the new goal and current weight. Show the user the new targets and confirm. Optionally: prompt to regenerate the programme ("Your goal changed — rebuild your 4-week programme for muscle building?"). This is a 30-minute feature that makes goal changes meaningful rather than cosmetic.

The 5 fixes that matter most
1
Day 28 completion + new block generation
Prevents the highest-risk churn window: weeks 5–8
Not built
Why first
After the initial programme, the product has no lifecycle. On Day 29, Soma becomes a static plan that never changes. This is the moment most subscribers will decide whether to keep paying. Reassessment + new block generation = the feature that turns a one-month trial into a long-term subscriber.
What to build
On generatedAt + 28 days: intercept page load, show achievement screen with 28-day data summary, prompt updated weight/check-in, generate new 4-week block using updated profile. Estimated build: 4–6 hours.
2
Apply weekly reconciliation adaptations
The AI produces the right answer — it just doesn't apply it
Not wired
Why second
The reconciliation already generates structured adaptations[] JSON from Claude. A "Apply to next week?" confirm button is all that's needed to make Soma genuinely adaptive. Without it, the intelligence is invisible — users never see the system respond to their actual behaviour.
What to build
Read profile.weeklyAdaptation.adaptations[]. For each adaptation, generate a week override. Show before/after comparison. One confirm. Estimated build: 2–3 hours.
3
Sleep-adaptive intensity on page load
The single feature that makes the daily plan feel alive
Built ✓
Status
Shipped. checkSleepIntensityBanner() runs on every training day load. Reads bl_daylog_[YESTERDAY].sleepActual. If <6h and intensity not already reduced: amber banner appears with one-tap "Reduce to Foundation intensity →". applySleepIntensityReduction() opens the session editor at beginner scheme. Seen-once per day via bl_sleep_banner_[DATE] key.
What it enables
Sleep is now logged in 3 places (EOD section, 8pm evening prompt bar, day-close card) to maximise data availability for the next morning's check.
4
Travel mode / Today I'm… status chips
Covers travelling and time-short — IBS, sick, sore still outstanding
Partial — 2 of 5 chips built
Built
Travelling — full travel mode with bodyweight session swap (S3 above). Short on time — time selector (30/45/60/90 min) + contextual "⏱ Short on time?" chip (S4 above).
Not yet built
Gut flare (low-FODMAP meal swap), Tired / low energy (reduce intensity without requiring sleep data), Sore / DOMS (active recovery overlay). These three cover scenarios N4, and the edge cases where sleep data wasn't logged the night before.
What to build next
Add [Tired] [Gut flare] [Sore] chips to the mission header chip row. Each triggers a day-scoped override — clears at midnight. The travel and short-on-time architecture (day-scoped, chip-triggered, plan-mutating) is the right pattern — extend it to these 3 remaining states.
5
Profile change → plan cache bust
1-line fix, closes a confusing inconsistency
1-line fix
Why fifth
When a user updates their injury or goal in the profile editor, they expect the plan to change. Right now it doesn't — the cached plan ignores the updated profile until manually rebuilt. This is confusing and erodes confidence in the app's responsiveness.
What to build
In saveProfile(p) in bodylens-profile.html, after writing to localStorage, add: localStorage.removeItem('dayplan_v6r3_' + new Date().toISOString().slice(0,10)). That's it. Next time they open the daily plan, it regenerates with the updated profile. Estimated build: 5 minutes.
