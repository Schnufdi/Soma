Audit methodology

This backtest was performed by reading every HTML and JS file in the project, mapping all localStorage reads/writes, tracing navigation paths across all three nav surfaces (desktop, vault dropdown, mobile menu), and cross-referencing profile field names between onboarding and consuming pages. No assumptions — everything is sourced from the actual code.

Critical issues — fix before launch
day.html vs dailyplan.html — split entry point
Critical
Onboarding (bodylens-onboard.html line 1147) redirects new users to bodylens-day.html after programme generation. But the main nav's "Today" link and brand logo both point to bodylens-dailyplan.html. These are different pages with different feature sets. New users land on a stripped-down view and can't find the full plan editor — they'd have to discover dailyplan.html independently.
Redirect day.html to dailyplan.html, or update onboarding to land on dailyplan.html directly.
bodylens-pro.html — linked everywhere, doesn't exist
Critical
The mobile menu "Upgrade" section links to /bodylens-pro.html. The page does not exist. Any user who taps the upgrade link gets a 404. This is a broken conversion path for your most intent-signalling action.
Create a simple pro.html paywall/upgrade page, or remove the nav link until Stripe is live.
bodylens-profiles.html — vault link leads nowhere
Critical
"Profile Vault" in the vault dropdown links to /bodylens-profiles.html, which doesn't exist. Users clicking this in the vault (likely founders or early users exploring) hit a 404. Erodes trust in the product's polish.
Create the page (could be a profile summary view) or remove the link.

The core user journey — mapped

The intended loop is: onboarding → programme → daily plan → check-in → weekly review → updated programme. This loop is partially wired. The onboarding → daily plan connection has a critical split, and the check-in → programme update feedback loop is not yet complete.

Missing feedback loops
Check-in doesn't update the programme
High
bodylens-checkin.html writes reflection data to bl_daylog_* but there's no mechanism that reads this and adapts the weekPlan or programme. The weekly reconciliation / adaptation loop — the product's core intelligent feedback mechanism — is architecturally defined (weeklyAdaptation field exists in bl_profile) but not wired end-to-end. This is the most important unbuilt feature — without it, Soma is a static programme generator, not a learning system.
Wire checkin → synthesis.html → weeklyAdaptation → programme update. bodylens-synthesis.html may already do this — it just needs to be in the main flow.
Profile has no edit interface
High
bodylens-profile.html is read-only. Users who complete onboarding and then change — lose weight, develop an injury, change training days — cannot update their profile short of resetting and redoing onboarding. This will be a support issue and a churn driver. People's lives change; the programme needs to reflect that without restarting from scratch.
Build basic profile edit for key fields: weight, training days, health conditions, injury status. Doesn't need to be full onboarding redux — just a settings page.
Journey strip "Strategy" step depends on weekPlan length
Medium
nav.js marks "Strategy" complete only when p.weekPlan && p.weekPlan.length >= 5. But weekPlan should contain exactly 7 entries (one per day). If AI generation returns a partial plan — say, only 5 training days and no rest day entries — the condition passes but the programme is incomplete. Conversely, if generation fails and stores an empty array, the user's journey strip shows "Strategy incomplete" even after they've done onboarding.
Change condition to check for programme generation timestamp: p.generatedAt — a more reliable signal than array length.

localStorage schema — key inventory

70+ localStorage keys in use. The schema is functional but undocumented. Some keys are date-scoped, some are persistent — there's no expiry or cleanup mechanism.

Key categories and patterns
  • Profile (persistent): bl_profile — master record, entire user object
  • Day logs (date-scoped): bl_daylog_YYYY-MM-DD — daily activity records
  • Macros (date-scoped): bl_macros_YYYY-MM-DD — daily macro targets
  • Weekly (week-scoped): bl_weekledger_YYYY-WW — weekly summaries
  • Form backup: bl_formdata_backup — onboarding form state
  • Prep/fridge: bl_prep_*, bl_fridge_* — meal prep tracking
  • No cleanup mechanism — date-scoped keys accumulate indefinitely. A user active for 1 year generates 365+ daylog keys. No TTL, no pruning. Will eventually hit localStorage quota (~5MB).
  • Cache keys: dayplan_v6r3_* — version-stamped but if version changes orphaned cache entries are never cleared.
Coach integration — what works
01
Risk taxonomy injection works correctly
risk-classifier.js correctly scans healthConditions and injuries, matches against RISK_TAXONOMY, and injects mandatory constraints into the coach system prompt. IBS, osteoporosis, cardiovascular, and other entries are wired. The coach cannot ignore these constraints — they're injected at the system level, not as user messages.
02
Page-aware context switching works
Coach.js reads the meta[name="bl-page"] tag and adjusts its context accordingly. On food/fuel pages it injects cooking preferences. On science pages it adds the relevant science context. This means the coach answers differently on the sleep page vs the training page — which is the right behaviour.
03
Graceful degradation is thorough
Every profile field access in coach.js uses || fallbacks or conditional checks. A brand new user with a minimal profile gets a functional (if less personalised) coach response — they're never shown an error or a blank response due to a missing field.

Branding — Soma vs BodyLens
01
Page titles: migrated
All <title> tags say "Soma —". All visible hero/brand text says "Soma." PWA manifest name is "Soma." This is correct and consistent.
02
File names: not migrated (bl_ prefix)
All HTML files are still named bodylens-*.html. All localStorage keys use the bl_ prefix. This is a cosmetic issue for users (URLs contain "bodylens"), a storage issue for brand cohesion in browser dev tools, and a maintenance issue for the codebase. A full rename would require updating every internal link across every page — high effort, low urgency.

Coach references: mostly migrated
coach.js refers to "Soma Coach" in user-visible strings. A few code comments still say "BodyLens" — invisible to users, fine to clean up gradually.
Content gap: profile editing
No way to update your profile without resetting
High
bodylens-profile.html displays profile data but has no edit functionality. The only way to change your weight, training days, health conditions, or goals is to use the Reset function (bodylens-reset.html) which wipes the entire profile — including all day logs, weekly data, and programme history.

Real-world scenario: A user loses 8kg and wants to update their weight. They can't. A user develops a new knee injury mid-programme. They can't add it. A user starts experiencing IBS symptoms. They can't add it to health conditions. The coach and daily plan remain calibrated to their old state indefinitely.
Build a lightweight profile editor for weight (weekly update), health conditions, injury status, and training days. These are the four most likely to change.
